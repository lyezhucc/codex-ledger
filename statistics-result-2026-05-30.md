## 核心算法

Codex 的 `token_count` 事件中，`total_token_usage` 是**会话累计值**，不能直接相加，否则会严重高估。

**算法**：在同一个 rollout 文件中，按时间顺序对相邻 token_count 事件做差值：

```
delta = current_total_token_usage - previous_total_token_usage
```

- `delta > 0` → 正常增量，计入
- `delta == 0` → 重复广播（rate limit 更新），跳过
- `delta < 0` → 计数器重置，用 current 值作为新一段增量，记录 warning

**验证用例**：
```
第 1 条: total = 1000
第 2 条: total = 1500
第 3 条: total = 1500

正确: 1000 + 500 + 0 = 1500
错误: 1000 + 1500 + 1500 = 4000
```

## 官网对账建议

1. **先确认官网显示的口径** → 是 raw total_tokens 还是 billable-equivalent / credits
2. **用 `--exclude-suspicious-first-baseline` 模式** → 排除 fork 继承历史
3. **查看 `suspicious_events.csv`** → 如果 >0 条且数值大，就是主要偏差来源
4. **如果官网数字接近 2.46 亿** → 官网用的是非缓存折算口径
5. **如果官网数字接近 25.7 亿** → 官网用的是 raw total_tokens 口径
6. **如果都不接近** → 检查是否有其他 Codex 配置目录未扫描、Codex Web/App 用量不写本地日志

## 统计口径说明

1. **total_token_usage 是累计值**，同一个 rollout 内必须做 delta，不能直接相加
2. **cached_input_tokens** 是 input_tokens 的子集，不重复计入 total_tokens
3. **reasoning_output_tokens** 是 output_tokens 的明细字段，不重复计入
4. 日期的划分按 `Asia/Shanghai` 时区，基于每条事件的时间戳
5. 跨 `--since` 边界时：`--since` 之前的事件不产生增量，但会推进 `prev_tokens` 基线

## 已知局限

1. 只统计了 `~/.codex` 目录
2. Codex Web / App 的远程任务如果不写本地 rollout 日志，统计不到
3. `codex exec --json` 等 headless 输出可能不走普通 session 目录
4. Fork/resume session 首条大累计值：当前 `--since 2026-05-03` 范围内无此类事件，但跨更大时间范围时可能出现。使用 `--exclude-suspicious-first-baseline` 可排除
