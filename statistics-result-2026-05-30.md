## 核心算法

Codex 的 `token_count` 事件中，`total_token_usage` 是**会话累计值**，不能直接相加，否则会严重高估。

**算法**：在同一个 rollout 文件中，按时间顺序对相邻 token_count 事件做差值：

```
delta = current_total_token_usage - previous_total_token_usage
```

- `delta > 0` → 正常增量，计入
- `delta == 0` → 重复广播（rate limit 更新），跳过
- `delta < 0` → 计数器重置，用 current 值作为新一段增量，记录 warning

**验证用例**：
```
第 1 条: total = 1000
第 2 条: total = 1500
第 3 条: total = 1500

正确: 1000 + 500 + 0 = 1500
错误: 1000 + 1500 + 1500 = 4000
```

## 统计口径说明

1. **total_token_usage 是累计值**，同一个 rollout 内必须做 delta，不能直接相加
2. **cached_input_tokens** 是 input_tokens 的子集，不重复计入 total_tokens
3. **reasoning_output_tokens** 是 output_tokens 的明细字段，不重复计入
4. 日期的划分按 `Asia/Shanghai` 时区，基于每条事件的时间戳
5. 如果一个 rollout 文件跨越多天，事件会被拆分到各自的日期
6. 如果文件中的 `timestamp` 字段缺失/解析失败，尝试从文件名推断时间
7. 没有 `timestamp` 解析失败或日期超出统计范围的 token_count 事件会被跳过
8. 跨 `--since` 边界时：`--since` 之前的事件不产生增量，但会推进 `prev_tokens` 基线

为确保算法正确性，进行了独立 delta 计算验证：

```
token_count 事件总数:      38,066
since 范围内的事件数:      22,947
delta 总和 (全部, 无过滤): 3,577,490,985
delta 总和 (since 过滤后): 2,579,861,626
脚本输出:                  2,579,861,626
差异:                      0 ✅
```

- 未过滤的 delta 总和 (35.77 亿) 与过滤后 (25.80 亿) 相差 9.98 亿，来自 `--since 2026-05-02` 之前的 3-4 月事件被正确排除
- 跨 `--since` 边界的 `prev_tokens` 基线维护正确：边界前的事件推进基线但不产生增量，边界后的事件正常做 delta
- 两个完全独立的实现得到相同结果，算法正确性已验证

## 已知局限

1. 只统计了 `~/.codex` 目录，如果有多个 Codex 账号/配置目录未传入，不会被计入
2. Codex Web / App 的远程任务如果不写本地 rollout 日志，统计不到
3. `codex exec --json` 等 headless 输出可能不走普通 session 目录
4. Fork/resume session 如果继承了历史累计值，第二个 token_count 开始会被 delta 算法自动消除；但如果是第一个 token_count 就继承了大量历史（超过 500万阈值），会被标记为 suspicious inherited baseline，仍会产生一个"首次增量 = 全部累计值"的事件——这时数据会高估
